## 🚀 About This Organization

**Brain Crafters** is a collaborative learning organization for aspiring Software Development Engineers (SDE1) focused on **Web Development**. We are Cohort 3C — a tight-knit group of developers building real-world skills through hands-on projects, peer collaboration, and continuous learning.

Our mission is to grow from learners into job-ready web developers by working together, sharing knowledge, and shipping meaningful projects.

## 🗺️ Getting Started (New Member Onboarding)

Welcome to the team! Follow these steps to get up and running:

### 1. Set Up Your GitHub Profile
- Make sure your GitHub profile has your **real name** and a **profile photo** so teammates can recognize you.
- Add a short bio describing your background and what you're learning.

### 2. Accept the Organization Invitation
- Check your email for an invitation to join **3C-SDE1-BRAIN-CRAFTERS-WD**.
- Accept the invite and explore the repositories listed under the organization.

### 3. Introduce Yourself
- Open a new **Discussion** or **Issue** in this repository titled `[Intro] Your Name` and tell the team:
  - Where you're from
  - What excites you about web development
  - Your favorite tool or technology so far

### 4. Set Up Your Local Development Environment
Make sure you have the following installed:

| Tool | Purpose | Link |
|------|---------|-------|
| **Git** | Version control | [git-scm.com](https://git-scm.com) |
| **Node.js (LTS)** | JavaScript runtime | [nodejs.org](https://nodejs.org) |
| **VS Code** | Recommended code editor | [code.visualstudio.com](https://code.visualstudio.com) |
| **GitHub CLI** | Interact with GitHub from the terminal | [cli.github.com](https://cli.github.com) |

### 5. Clone Your First Repository
```bash
# Clone a repository from the organization
git clone https://github.com/3C-SDE1-BRAIN-CRAFTERS-WD/<repo-name>.git
cd <repo-name>

# Install dependencies (if applicable)
npm install
```

### 6. Understand Our Git Workflow
We follow a **feature branch workflow**:

```
main          ← production-ready code
  └── dev     ← integration branch
        └── feature/your-feature-name   ← your work
```

1. Always branch off from `dev`.
2. Name your branch descriptively: `feature/login-page`, `fix/navbar-bug`, `chore/update-readme`.
3. Open a **Pull Request (PR)** into `dev` when your feature is ready.
4. Request at least **one peer review** before merging.
5. Keep commits small and focused — one logical change per commit.

## 📋 Coding Standards & Best Practices

To keep our codebase clean and consistent:

- ✅ Write **meaningful commit messages** (e.g., `feat: add user authentication`, `fix: resolve button alignment issue`).
- ✅ Follow the **[Conventional Commits](https://www.conventionalcommits.org/)** standard.
- ✅ Comment your code where logic is non-obvious.
- ✅ Use **camelCase** for variables and functions, **PascalCase** for components and classes.
- ✅ Review your own PR diff before requesting reviews.
- ✅ Write or update tests when adding new features.
- ❌ Never commit directly to `main`.
- ❌ Never push sensitive data (API keys, passwords, `.env` files).
